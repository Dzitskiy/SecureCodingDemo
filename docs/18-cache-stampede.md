# Cache Stampede

[← К оглавлению](../README.md#topics)

## Что не так

Когда горячий ключ истекает одновременно для множества запросов, все они пытаются сразу восстановить одно и то же значение из БД или другого медленного источника.

## Типовой уязвимый кейс

Обычный шаблон `cache-aside` без координации: после истечения времени жизни каждый запрос повторяет дорогую пересборку.

## Пример атаки

```text
50 parallel requests right after cache expiry
```

## Последствия

- Перегрузка БД.
- Всплески задержек.
- Каскадные сбои в соседних сервисах и зависимостях.

## Как исправлять

- Использовать блокировку одиночного пересчета для восстановления записи кеша.
- Применять общий кеш вроде `Redis` для повторного использования между экземплярами.
- Использовать устаревшее значение на время обновления и добавлять случайный разброс к времени жизни.

## Контрольный список

- Делать предварительный прогрев для горячих ключей.
- Собирать метрики по всплескам промахов кеша.
- Проводить тестирование на синхронное истечение срока жизни.
