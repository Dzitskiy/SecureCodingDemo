# Broken Access Control

## Что не так

Broken Access Control возникает, когда приложение знает пользователя, но не проверяет, имеет ли он право выполнить конкретное действие или получить конкретный объект.

## Типовой уязвимый кейс

API возвращает заказ по `orderId`, не проверяя владельца заказа. Атакующий перебирает идентификаторы и получает чужие данные.

## Payload

```text
GET /api/orders/unsafe/103
```

## Последствия

- Horizontal и vertical privilege escalation.
- Утечка персональных, финансовых и бизнес-данных.
- Изменение чужих объектов через прямой доступ к идентификаторам.

## Как исправлять

- Проверять authorization на уровне business operation, а не только на уровне маршрута.
- Проверять ownership каждого объекта перед чтением или изменением.
- Не доверять `userId`, `role`, `tenantId` и другим access-control полям из request body.

## Production checklist

- Policy-based authorization для критичных операций.
- Негативные тесты: пользователь A не видит и не меняет объекты пользователя B.
- Audit отказов доступа и подозрительного перебора идентификаторов.
