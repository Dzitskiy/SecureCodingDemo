# SSRF

[← К оглавлению](../README.md#topics)

## Что не так

`SSRF` позволяет использовать сервер как доверенный сетевой прокси. Через него атакующий получает доступ к внутренним сервисам и конечным точкам с метаданными облака.

## Типовой уязвимый кейс

API принимает произвольный `URL` и без дополнительной проверки скачивает его от имени сервера.

## Пример атаки

```text
http://169.254.169.254/latest/meta-data/
```

## Последствия

- Доступ к внутренним административным панелям и сервисам.
- Утечка облачных учетных данных и метаданных экземпляра.
- Дальнейшее продвижение внутрь приватной сети.

## Как исправлять

- Разрешать только список допустимых имен узлов и `HTTPS`.
- После разрешения имени блокировать локальные и приватные диапазоны адресов.
- Ограничивать перенаправления, размер ответа и тайм-аут.

## Контрольный список

- Использовать исходящий межсетевой экран или прокси.
- Закрыть метаданные облака через `IMDSv2` или аналогичные политики.
- Мониторить подозрительные исходящие назначения.
