# Security Misconfiguration

[← К оглавлению](../README.md#topics)

## Что не так

Небезопасная конфигурация появляется, когда приложение запускается с рискованными настройками: подробные ошибки наружу, отсутствующие защитные заголовки, открытый режим отладки, слишком широкие политики `CORS` или `CSP`, а также лишние служебные конечные точки.

## Типовой уязвимый кейс

API возвращает клиенту имя окружения, трассировку стека и внутренние настройки, а `HTTP`-ответ не содержит базовых защитных заголовков.

## Пример атаки

```text
GET /api/configuration/unsafe-headers
```

## Последствия

- Утечка путей, версий, окружения и внутренней структуры приложения.
- Упрощение атак на `XSS`, внедрение во фреймы и подмену типа содержимого.
- Повышенный риск эксплуатации из-за включенных режимов отладки или настроек по умолчанию.

## Как исправлять

- Не возвращать трассировку стека и внутренние настройки внешним клиентам.
- Добавлять базовые защитные заголовки: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `CSP`.
- Разделять конфигурацию для разработки и промышленной эксплуатации.

## Контрольный список

- Устанавливать `ASPNETCORE_ENVIRONMENT=Production` в промышленной среде.
- Закрывать или защищать `Swagger`, отладочные конечные точки и подробные ошибки.
- Проверять `CORS`, `CSP`, cookie и заголовки на ревью безопасности перед релизом.
