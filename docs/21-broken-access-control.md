# Broken Access Control

[← К оглавлению](../README.md#topics)

## Что не так

Нарушение контроля доступа возникает, когда приложение знает пользователя, но не проверяет, имеет ли он право выполнить конкретное действие или получить конкретный объект.

## Типовой уязвимый кейс

API возвращает заказ по `orderId`, не проверяя владельца заказа. Атакующий перебирает идентификаторы и получает чужие данные.

## Пример атаки

```text
GET /api/orders/unsafe/103
```

## Последствия

- Горизонтальное и вертикальное повышение привилегий.
- Утечка персональных, финансовых и прикладных данных.
- Изменение чужих объектов через прямой доступ к идентификаторам.

## Как исправлять

- Проверять авторизацию на уровне прикладной операции, а не только на уровне маршрута.
- Проверять принадлежность каждого объекта перед чтением или изменением.
- Не доверять `userId`, `role`, `tenantId` и другим полям контроля доступа из тела запроса.

## Контрольный список

- Настроить авторизацию на основе политик для критичных операций.
- Добавить негативные тесты: пользователь A не видит и не меняет объекты пользователя B.
- Проводить аудит отказов доступа и подозрительного перебора идентификаторов.
