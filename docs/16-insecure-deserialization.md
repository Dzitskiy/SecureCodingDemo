# Insecure Deserialization

[← К оглавлению](../README.md#topics)

## Что не так

Если клиент выбирает конкретный тип для десериализации, он влияет на то, какие объекты создаются внутри приложения и какой код затем выполняется в конвейере обработки.

## Типовой уязвимый кейс

API принимает `type` и `payload`, после чего пытается создать произвольный тип `.NET` по имени.

## Пример атаки

```json
{"type":"SecureCodingDemo.Modules.GrantAdminAction","payload":{"UserId":"1"}}
```

## Последствия

- Обход прикладных правил через неожиданные типы-команды.
- Использование цепочек опасных объектов и побочных эффектов конструкторов или сеттеров.
- Повышение привилегий и нарушение проверки рабочего процесса.

## Как исправлять

- Не принимать имя типа от клиента.
- Десериализовать только в известные `DTO`.
- Если полиморфизм действительно нужен, вводить короткие дискриминаторы из списка допустимых значений.

## Контрольный список

- Валидировать схему на входе.
- Проверять настройки сериализатора в общих библиотеках.
- Добавить отдельные тесты на небезопасный полиморфизм.
