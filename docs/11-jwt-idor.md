# JWT / IDOR

[← К оглавлению](../README.md#topics)

## Что не так

`JWT` подтверждает личность пользователя, но не выдает автоматически право на любой объект. Если API возвращает запись только по `id`, появляется `IDOR`.

## Типовой уязвимый кейс

Пользователь с токеном для `userId=1` запрашивает заказ другого пользователя по известному идентификатору.

## Пример атаки

```text
GET /api/orders/unsafe/103
```

## Последствия

- Горизонтальное повышение привилегий.
- Утечка персональных и прикладных данных.
- Использование устаревших ролей из старых маркеров доступа.

## Как исправлять

- Проверять принадлежность каждого объекта.
- Критичные утверждения перепроверять по актуальному серверному источнику данных.
- Делать маркеры доступа короткоживущими.

## Контрольный список

- Настроить авторизацию на уровне прикладной операции.
- Продумать стратегию отзыва доступа после смены роли.
- Проводить аудит отказов доступа и шаблонов перебора идентификаторов.
