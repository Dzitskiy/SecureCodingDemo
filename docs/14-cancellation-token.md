# CancellationToken

[← К оглавлению](../README.md#topics)

## Что не так

Если сервер игнорирует отмену запроса, он продолжает занимать процессор, сокеты и соединения с БД даже после разрыва связи клиентом или истечения времени ожидания на шлюзе.

## Типовой уязвимый кейс

Долгая конечная точка делает `Task.Delay`, обращения к БД и внешние `HTTP`-запросы без передачи `CancellationToken`.

## Пример атаки

```text
/api/cancellation/unsafe?seconds=120
```

## Последствия

- Накопление бесполезной работы.
- Истощение пула потоков и пула соединений.
- Рост задержек для нормальных запросов.

## Как исправлять

- Передавать `CancellationToken` во все асинхронные API.
- Создавать связанный маркер с серверным бюджетом времени.
- Проверять отмену внутри собственных циклов.

## Контрольный список

- Определить бюджет времени для каждого типа операции.
- Собирать метрики по отмененным и завершенным запросам.
- Проверять все фоновые циклы внутри конвейера обработки запроса.
