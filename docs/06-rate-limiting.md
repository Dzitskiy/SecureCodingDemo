# Rate Limiting

[← К оглавлению](../README.md#topics)

## Что не так

Без ограничений вход в систему и дорогие конечные точки становятся удобной целью для перебора паролей и атак на отказ в обслуживании. Даже неуспешные запросы тратят процессор, соединения с БД и место в журналах.

## Типовой уязвимый кейс

Вход принимает бесконечное число попыток с одного IP-адреса или для одного пользователя без замедления и временной блокировки.

## Пример атаки

```json
{"username":"admin","password":"guess"}
```

## Последствия

- Массовый перебор учетных данных.
- Давление на учетные записи с целью захвата.
- Деградация API при всплесках запросов.

## Как исправлять

- Применять ограничители с фиксированным окном, скользящим окном или ведром токенов.
- Ограничивать не только по IP-адресу, но и по пользователю или ключу клиента.
- Комбинировать с многофакторной аутентификацией, защитой от ботов и журналированием сигналов атаки.

## Контрольный список

- Задать отдельные лимиты для входа, поиска и тяжелых отчетов.
- Настроить оповещения на рост `429` и аномальные шаблоны.
- Добавить репутационные блокировки и прогрессивные задержки.
