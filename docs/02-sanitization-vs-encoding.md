# Sanitization vs Encoding

[← К оглавлению](../README.md#topics)

## Что не так

Кодирование и очистка HTML решают разные задачи. Кодирование нужно почти всегда при выводе недоверенных данных. Очистка нужна только тогда, когда приложение сознательно разрешает ограниченный HTML.

## Типовой уязвимый кейс

Разработчик удаляет только `<script>` через регулярное выражение и считает HTML безопасным, хотя полезная нагрузка может использовать `SVG`, ссылки с `javascript:` или встроенные обработчики событий.

## Пример атаки

```html
<svg><a xlink:href="javascript:alert(1)">click</a></svg>
```

## Последствия

- Сохраненный `XSS` при хранении якобы очищенного HTML.
- Рендеринг активного содержимого вместо текста.
- Быстрое отставание самодельного очистителя от реальных техник обхода.

## Как исправлять

- Обычный текст всегда кодировать.
- Ограниченный HTML очищать только зрелой библиотекой со списком допустимых тегов и атрибутов.
- Не считать, что очистка HTML заменяет контекстное кодирование вывода.

## Контрольный список

- Зафиксировать единую политику для ограниченного HTML.
- Добавить отдельные тесты для `SVG`, `data:` и встроенных обработчиков.
- Задокументировать список разрешенных тегов и атрибутов.
