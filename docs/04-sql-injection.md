# SQL Injection

[← К оглавлению](../README.md#topics)

## Что не так

Если пользовательский ввод попадает в SQL как часть синтаксиса, атакующий может менять условие `WHERE`, добавлять `UNION`, влиять на сортировку и даже выполнять разрушающие команды.

## Типовой уязвимый кейс

Запрос строится строковой интерполяцией: `... where username = '{username}'`.

## Пример атаки

```sql
' OR '1'='1
```

## Последствия

- Обход аутентификации.
- Утечка данных через объединение запросов и слепые техники.
- Изменение или удаление данных.

## Как исправлять

- Использовать параметризованные запросы.
- Ограничивать права учетной записи БД.
- Для динамических структурных частей применять список допустимых значений, а не строковую конкатенацию.

## Контрольный список

- Добавить проверки `CI` на опасные места с незащищенным SQL.
- Выдавать права только на чтение там, где запись не нужна.
- Мониторить аномальные шаблоны SQL и ошибки разбора запросов.
